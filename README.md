Прототип browser automation-агента на Python + Playwright + OpenAI-compatible LLM.

## Что это по факту

Это **не готовый production-агент** и не полноценная реализация всего ТЗ. Текущее состояние проекта — **исследовательский прототип** с базовым циклом:

`observe -> plan -> act`

Агент:
- открывает **видимый Chromium** через Playwright;
- снимает упрощённое наблюдение страницы;
- отправляет сжатый контекст в LLM;
- получает JSON-план следующего шага;
- выполняет один из базовых инструментов;
- пишет подробные логи в терминал.

Проект подходит как демонстрация архитектуры автономного браузерного агента, но **не гарантирует стабильное выполнение сложных end-to-end пользовательских сценариев** на внешних сайтах.

## Что в проекте реально реализовано

### 1. Видимый браузер через Playwright
- По умолчанию Chromium запускается **не в headless**.
- Есть поддержка `--profile-dir` для persistent browser profile.
- Можно запускать ручной логин через `--pause-for-login`.

### 2. Базовый observe-plan-act loop
- `main.py` запускает цикл агента.
- `browser.py` собирает наблюдение:
  - URL
  - title
  - ARIA snapshot
  - обобщённый DOM scan
- `planner.py` просит LLM вернуть **JSON с одним или несколькими шагами**.
- `executor.py` исполняет действия.

### 3. Набор инструментов
Поддерживаются инструменты:
- `navigate`
- `click`
- `type`
- `scroll`
- `wait`
- `back`
- `ask_user`
- `finish_task`

### 4. Память и контекст
- В `memory.py` хранится краткая история:
  - посещённые страницы
  - успешные шаги
  - ошибки
  - action history
- Контекст в модель отправляется в **сжатом виде**, а не как сырая страница целиком.

### 5. Защита от некоторых опасных кликов
- В `executor.py` есть простая эвристика для потенциально деструктивных кликов.
- Подозрительные клики запрашивают подтверждение в терминале.
- Это **эвристика по ключевым словам**, а не полноценная policy/guardrail система.

### 6. Error reflector
- После неудачного действия может вызываться `error_reflector.py`.
- Он просит LLM дать короткую подсказку для recovery.

### 7. Smoke test
- Есть `smoke_test.py`, который проверяет базовые действия без внешнего сайта и без полноценного сценария с LLM-логикой.

## Что НЕ реализовано или работает нестабильно

### 1. Нет надёжного выполнения сложных пользовательских сценариев
На текущем этапе агент **может зависать, ходить по кругу, повторять действия или выбирать нерелевантные элементы** на реальных сайтах.

Особенно это заметно на сложных динамических сайтах вроде hh.ru.

### 2. Нет гарантированного соответствия ТЗ “под ключ”
Если ТЗ подразумевает стабильный сценарий уровня:
- открыть hh.ru,
- изучить профиль,
- найти релевантные вакансии,
- извлечь ключевую информацию,
- подготовить персонализированные письма,
- остановиться перед финальной отправкой,

то **текущая версия проекта это не гарантирует**.

### 3. Нет site-specific orchestration
Проект почти полностью опирается на:
- ARIA / accessible names,
- общий DOM scan,
- LLM-планирование.

То есть здесь **нет глубокой site-specific логики**, которая бы надёжно фиксировала:
- как искать именно резюме,
- как отличать рекламу от полезного блока,
- как детерминированно собирать вакансии,
- как не заходить в циклы между страницами.

### 4. JSON-зависимость от качества модели
Вся планирующая часть сильно зависит от того, насколько локальная модель:
- понимает промпт,
- соблюдает JSON-схему,
- не обрезает ответ,
- не уходит в странные повторяющиеся решения.

Если модель слабо держит структуру, агент деградирует.

### 5. Нет гарантированной устойчивости на SPA / lazy-loaded интерфейсах
Сайт с динамической подгрузкой контента может приводить к таким проблемам:
- элемент ещё не появился;
- accessible name нестабилен;
- клик уходит не туда;
- поле ввода не находится;
- агент повторяет один и тот же шаг.

### 6. Security layer ограничен
Защита от опасных действий — это **не полноценный sandbox** и не строгая система разрешений.
Это просто словарная эвристика по названиям кнопок/ссылок.

## Для чего проект реально подходит

Проект можно адекватно позиционировать как:

- **прототип автономного browser agent-а**;
- **демо архитектуры observe-plan-act**;
- **учебный/экспериментальный проект** по связке Playwright + LLM;
- база для дальнейшей доработки под конкретный сайт или узкий сценарий.

## Для чего проект пока не подходит

Не стоит честно позиционировать текущую версию как:
- готовый AI job-apply agent;
- надёжный autonomous assistant для hh.ru;
- production-ready browser copilot;
- полностью завершённое решение по сложному тестовому сценарию.

## Архитектура проекта

```text
browser_agent/
  agent/
    main.py             # основной цикл observe -> plan -> act
    browser.py          # Playwright + наблюдение страницы
    planner.py          # формирование запроса к LLM и чтение JSON-плана
    llm.py              # OpenAI-compatible клиент
    executor.py         # выполнение инструментов
    memory.py           # краткая память шага/истории
    error_reflector.py  # recovery hints после ошибок
    tools.py            # dataclasses для action / observation / results
    security_keywords.py# эвристики для подозрительных кликов
```

## Как запускать

### Установка

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
playwright install chromium
```

### Настройка `.env`

Пример лежит в `.env.example`.
Нужен OpenAI-compatible endpoint, например LM Studio:

```env
LM_BASE_URL=http://127.0.0.1:1234/v1
LM_API_KEY=lm-studio
LM_MODEL=qwen2.5-7b-instruct
```

### Smoke test

```bash
python smoke_test.py
```

### Базовый запуск агента

```bash
python -m browser_agent.agent.main --verbose --task "Open a simple website and summarize it" --start-url "https://playwright.dev"
```

### С persistent profile

```bash
python -m browser_agent.agent.main --verbose --start-url "https://example.com" --profile-dir ".profiles/demo" --pause-for-login --task "..."
```

## Честная оценка готовности

Состояние проекта на момент этой версии:

- **Архитектура прототипа — есть**
- **Базовые инструменты — есть**
- **Интеграция Playwright + LLM — есть**
- **Логи и наблюдение — есть**
- **Надёжный end-to-end сценарий на реальном сложном сайте — нет**
- **Полное соответствие сложному тестовому ТЗ — нет**

Итоговая формулировка:

> Это рабочий исследовательский прототип browser automation-агента с LLM-планировщиком, но не завершённый production-ready продукт и не гарантированное решение сложного многошагового сценария на hh.ru.

## Ограничения

- Сильно зависит от качества локальной модели.
- Может ошибаться в выборе элементов.
- Может повторять шаги при слабой стратегии recovery.
- Плохо переносится на сложные динамические интерфейсы без дополнительной доработки.
- Не содержит надёжной site-specific логики для job application flows.
